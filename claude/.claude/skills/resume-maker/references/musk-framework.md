# MUSK framework

## what MUSK means

| component | weight | purpose |
|-----------|--------|---------|
| metrics | 40% | show the result |
| using action words | 15% | show ownership |
| stakeholders | 10% | show who benefited |
| keywords | 25% | match the role language |

The missing 10% is context, the why behind the work.

## three questions before writing a bullet

1. what changed
2. why it mattered
3. how it was done

## core bullet formulas

```text
[quantified result] by [key actions taken] using [skills or tools]
```

```text
[action + skills] [context], resulting in [quantified result]
```

Use both forms so the bullets do not sound repetitive.

## metrics come first

If there is no number, the bullet is probably weak.

Look for:

- time saved
- cost saved
- scale
- uptime or reliability
- faster response or delivery
- reduced manual work
- adoption by other teams

If no hard number exists, use a before-and-after state and flag the missing metric.

## strong verbs

Prefer:

- architected
- engineered
- implemented
- designed
- built
- automated
- deployed
- integrated
- migrated
- optimized

Avoid:

- helped
- assisted
- worked on
- was responsible for

## stakeholders

Make the impact human and concrete:

- users
- engineers
- the business
- other teams

## keywords

Pull keywords directly from the JD and mirror the employer's phrasing when it fits the evidence.

Common categories for infra roles:

- cloud
- containers
- IaC
- CI/CD
- observability
- data systems
- databases

## quick scoring rubric

Rate each bullet 1-5 on:

- metric strength
- action verb strength
- stakeholder clarity
- keyword relevance

If the overall score is below 3, rewrite it.
